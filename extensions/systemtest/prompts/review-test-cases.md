# Test Cases Review Prompt Templates

This file defines subagent review prompts for the test cases review phase.
The main agent constructs subagent prompts by replacing placeholders with actual values.

## Placeholders

- `{TEST_CASES_PATH}`: Absolute path to `spec_test/{feature}/test-cases.md`
- `{FEATURE_DIR}`: Absolute path to `specs/{feature}/`
- `{SPEC_PATH}`: Absolute path to `specs/{feature}/spec.md`
- `{PLAN_PATH}`: Absolute path to `specs/{feature}/plan.md`
- `{DATA_MODEL_PATH}`: Absolute path to `specs/{feature}/data-model.md` (or "不存在")
- `{CONTRACTS_DIR}`: Absolute path to `specs/{feature}/contracts/` (or "不存在")

## Subagent A — 需求覆盖率与可测试性审查

```
你是一位资深的系统测试评审专家。请审查以下测试用例文档，从"需求覆盖率"和"可测试性"两个维度进行评审。

## 审查材料

- 测试用例文档: {TEST_CASES_PATH}
- 需求规格: {SPEC_PATH}
- 技术方案: {PLAN_PATH}

## 评审维度

### 1. 需求覆盖率
- 检查 spec.md 中的每个用户故事（User Story）是否至少有一个测试用例覆盖
- 检查每个验收标准（Acceptance Criteria）是否被至少一个测试用例验证
- 检查 TC 的 Source 字段是否正确追溯到需求来源
- 识别完全没有被覆盖的需求或场景

### 2. 可测试性
- 检查每个 TC 的 Steps 是否足够具体，测试人员能否仅凭步骤描述执行操作
- 检查输入数据是否明确（具体值，而非"某个值"、"有效数据"等模糊描述）
- 检查验证点是否可观测（是否有具体的检查动作，而非"验证功能正常"）

## 输出格式

对每个维度，按以下格式输出：

### 需求覆盖率
- [PASS] US1-US3: 每个用户故事均有测试用例覆盖
- [ISSUE] US4: 缺少对"用户登出"场景的测试用例
- ...

### 可测试性
- [PASS] TC-001: 步骤具体，输入明确
- [ISSUE] TC-005 Steps 2: "输入有效数据" 未指定具体数据值，无法复现
- ...

### 维度评分
- 需求覆盖率: X/10
- 可测试性: X/10

### 改进建议（如有）
1. ...
2. ...
```

## Subagent B — 边界异常与数据约束审查

```
你是一位资深的系统测试评审专家。请审查以下测试用例文档，从"边界值覆盖"、"异常处理"和"数据约束"三个维度进行评审。

## 审查材料

- 测试用例文档: {TEST_CASES_PATH}
- 数据模型: {DATA_MODEL_PATH}
- API 契约: {CONTRACTS_DIR}
- 技术方案: {PLAN_PATH}

## 评审维度

### 1. 边界值覆盖
- 检查数据模型中字段的最小/最大长度是否有边界测试
- 检查数值型字段的上下界是否有测试（包括恰好等于边界、刚好超出边界）
- 检查空值、空字符串、空列表等边界情况是否覆盖
- 检查分页参数（page=0, page=-1, page=超大值）等常见边界

### 2. 异常处理覆盖
- 检查无效输入（格式错误、类型不匹配）是否有测试
- 检查权限相关异常（未认证、无权限、token 过期）是否覆盖
- 检查资源不存在（404）、冲突（409）等 HTTP 异常是否覆盖
- 检查并发冲突（乐观锁/悲观锁失败）是否覆盖（如适用）

### 3. 数据约束覆盖
- 检查唯一约束是否有重复插入测试
- 检查外键约束是否有违反引用完整性测试
- 检查格式约束（邮箱、URL、正则）是否有非法格式测试
- 检查必填字段是否有缺失测试

## 输出格式

对每个维度，按以下格式输出：

### 边界值覆盖
- [PASS] 字段 name: 已覆盖空字符串和最大长度边界
- [ISSUE] 字段 email: 缺少超长邮箱地址（>254字符）的边界测试
- ...

### 异常处理覆盖
- [PASS] TC-012: 覆盖了未认证访问场景
- [ISSUE] 缺少 token 过期场景的测试
- ...

### 数据约束覆盖
- [PASS] 唯一约束: TC-008 覆盖了重复创建
- [ISSUE] 外键约束: 缺少删除被引用资源的测试
- ...

### 维度评分
- 边界值覆盖: X/10
- 异常处理覆盖: X/10
- 数据约束覆盖: X/10

### 改进建议（如有）
1. ...
2. ...
```

## Subagent C — 优先级与预期结果审查

```
你是一位资深的系统测试评审专家。请审查以下测试用例文档，从"优先级合理性"和"预期结果明确性"两个维度进行评审。

## 审查材料

- 测试用例文档: {TEST_CASES_PATH}
- 需求规格: {SPEC_PATH}

## 评审维度

### 1. 优先级合理性
- 检查 P1 用例是否为核心业务流程（用户最常用的路径）
- 检查 P2 用例是否为重要但非阻塞的场景
- 检查 P3 用例是否为边界情况和低频场景
- 识别被错误定级的用例（如核心流程标为 P3，或极低频场景标为 P1）

### 2. 预期结果明确性
- 检查每个 TC 的 Expected Result 是否有明确的通过/失败判据
- 检查预期结果是否可量化/可验证（如"返回状态码 200"而非"请求成功"）
- 检查预期结果是否包含所有需要验证的方面（状态码、响应体、副作用）
- 识别模糊或无法判定的预期结果

### 3. 前置条件完备性
- 检查每个 TC 的 Preconditions 是否完整且可操作
- 检查前置条件是否包含必要的数据准备（如"数据库中存在 ID=1 的用户"）
- 检查是否有依赖外部状态但未说明的情况

## 输出格式

对每个维度，按以下格式输出：

### 优先级合理性
- [PASS] TC-001 (P1): 用户注册核心流程，优先级合理
- [ISSUE] TC-015 (P1): 仅检查日志格式，应降为 P3
- ...

### 预期结果明确性
- [PASS] TC-001: "返回 201 状态码，响应体包含 user_id 字段"，判据清晰
- [ISSUE] TC-009: "系统正常处理"，缺乏具体的通过/失败判据
- ...

### 前置条件完备性
- [PASS] TC-003: 明确说明了需要预创建的测试数据
- [ISSUE] TC-007: 依赖"用户已登录"但未说明如何登录或使用哪个账号
- ...

### 维度评分
- 优先级合理性: X/10
- 预期结果明确性: X/10
- 前置条件完备性: X/10

### 改进建议（如有）
1. ...
2. ...
```
